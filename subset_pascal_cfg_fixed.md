# Fixed Context-Free Grammar (CFG) for a Subset of Pascal

This is the fixed version of the original grammar, eliminating **left recursion** and **left factoring** issues for recursive-descent parsing.

---

## Issues Found & Fixed

### Left Recursion Eliminated:
1. `identifier_list` - direct left recursion
2. `declarations` - direct left recursion
3. `subprogram_declarations` - direct left recursion
4. `parameter_list` - direct left recursion
5. `statement_list` - direct left recursion
6. `simple_expression` - direct left recursion
7. `term` - direct left recursion

### Left Factoring Applied:
1. `variable` - common prefix `id`
2. `procedure_statement` - common prefix `id`
3. `factor` - common prefix `id`

## Fixed Productions

### Program structure

```
program  →  program id ( identifier_list ) ;
            declarations
            subprogram_declarations
            compound_statement
```

### Identifier list

```
identifier_list  →  id identifier_list'
identifier_list'  →  , id identifier_list'
                  |  ε
```

### Declarations

```
declarations  →  var identifier_list : type declarations
              |  ε
```

### Type

```
type  →  standard_type
      |  array [ num .. num ] of standard_type
```

### Standard type
Bottom-Up Parsing: Operator Precedence Parser
```
standard_type  →  integer
                |  real
```

### Subprogram declarations

```
subprogram_declarations  →  subprogram_declaration ; subprogram_declarations
                         |  ε
```

### Subprogram declaration

```
subprogram_declaration  →  subprogram_head declarations compound_statement
```

### Subprogram head

```
subprogram_head  →  function id arguments : standard_type ;
                  |  procedure id arguments ;
```

### Arguments

```
arguments  →  ( parameter_list )
            |  ε
```

### Parameter list

```
parameter_list  →  identifier_list : type parameter_list'
parameter_list'  →  ; identifier_list : type parameter_list'
                 |  ε
```

### Compound statement

```
compound_statement  →  begin
                      optional_statements
                      end
```

### Optional statements

```
optional_statements  →  statement_list
                      |  ε
```

### Statement list

```
statement_list  →  statement statement_list'
statement_list'  →  ; statement statement_list'
                 |  ε
```

### Statement

```
statement  →  variable assignop expression
           |  procedure_statement
           |  compound_statement
           |  if expression then statement else statement
           |  while expression do statement
```

> **Note (dangling-else):** Adding the production
> `statement → if expression then statement`
> introduces the classic "dangling-else" ambiguity, which can be resolved as discussed in Section 4.3 (or by shifting on shift/reduce conflicts in Yacc).

### Variable

```
variable  →  id variable'
variable'  →  [ expression ]
           |  ε
```

### Procedure statement

```
procedure_statement  →  id procedure_statement'
procedure_statement'  →  ( expression_list )
                      |  ε
```

### Expression list

```
expression_list  →  expression expression_list'
expression_list'  →  , expression expression_list'
                  |  ε
```

### Expression

```
expression  →  simple_expression
            |  simple_expression relop simple_expression
```

### Simple expression

```
simple_expression  →  term simple_expression'
                    |  sign term simple_expression'
simple_expression'  →  addop term simple_expression'
                    |  ε
```

### Term

```
term  →  factor term'
term'  →  mulop factor term'
      |  ε
```

### Factor

```
factor  →  id factor'
        |  num
        |  ( expression )
        |  not factor
factor'  →  ( expression_list )
        |  ε
```

> **Note:** There is no syntactic distinction between a simple variable and a call of a function without parameters. Both are generated by `factor → id factor'` with `factor' → ε`.

### Sign

```
sign  →  +
      |  -
```

## A.4 Lexical Conventions (Token Definitions) - Unchanged

1. Comments are surrounded by `{` and `}` and may not contain a `{`. They may appear after any token.
2. Blanks between tokens are optional, except keywords must be surrounded by blanks, newlines, the beginning of the program, or the final dot.
3. Keywords are reserved and appear in **boldface** in the grammar.

### Identifiers (`id`)

```
letter  →  [a-zA-Z]
digit   →  [0-9]
id      →  letter ( letter | digit )*
```

### Numbers (`num`) — unsigned integers/reals

```
digits             →  digit digit*
optional_fraction  →  . digits | ε
optional_exponent  →  ( E ( + | - | ε ) digits ) | ε
num                →  digits optional_fraction optional_exponent
```

### Operators

| Token      | Lexemes                              |
|------------|--------------------------------------|
| `relop`    | `=`, `<>`, `<`, `<=`, `>=`, `>`  (`<>` denotes ≠) |
| `addop`    | `+`, `-`, `or`                       |
| `mulop`    | `*`, `/`, `div`, `mod`, `and`        |
| `assignop` | `:=`                                 |

---

## Reserved Keywords (boldface terminals)

`program`, `var`, `array`, `of`, `integer`, `real`, `function`, `procedure`,
`begin`, `end`, `if`, `then`, `else`, `while`, `do`, `not`, `or`, `div`, `mod`, `and`.
